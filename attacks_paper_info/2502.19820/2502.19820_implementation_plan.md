## 2. Attack Overview
FITD is a multi-turn jailbreak attack that leverages the "foot-in-the-door" psychological effect. It starts with a benign version of the malicious query and gradually escalates the maliciousness over multiple turns. The attack employs two auxiliary mechanisms:
1.  **SlipperySlopeParaphrase (SSP)**: If the model refuses a query but the previous response was aligned, it generates an intermediate "bridge" query to smooth the transition.
2.  **Re-Align**: If the model's response to a previous query was misaligned (e.g., evasion or refusal), it explicitly prompts the model to correct its response to match the query's intent before proceeding.

The core innovation is the self-driven escalation where the model's own responses (commitments) make it susceptible to further malicious requests.

## 5. Data Flow and Control Flow
1.  **Initialization**:
    -   Generate `evolution_steps` (list of prompts) using `assistant_model`.
    -   Initialize `history` (messages list).
2.  **Main Loop** (`idx` from 0 to `steps-1`):
    -   `current_prompt = evolution_steps[idx]`
    -   **Interaction**:
        -   If `idx == 0`: Query target with `current_prompt`.
        -   If `idx > 0`: Call `_polish_content(prev_prompt, current_prompt, history)` -> Update history.
    -   **Check Response**:
        -   If `!is_refusal`: Continue to next step.
        -   If `is_refusal` (and `idx > 0`):
            -   Pop failed exchange.
            -   Check `_detect_alignment(prev_prompt, prev_response)`.
            -   **Branch 1 (Misaligned)**: Call `_realign`.
                -   Append Re-Align prompt.
                -   Query target.
                -   If successful (not refusal), continue.
            -   **Branch 2 (Aligned)**: Call `_ssp`.
                -   Generate `qmid` between `prev` and `current`.
                -   Try `_polish_content` with `qmid`.
                -   Retry up to 2 times.
                -   If successful, update `evolution_steps[idx-1]` to `qmid` (backtrack slightly) or insert `qmid`. (Repo implementation replaces `prompts[idx-1]` if `qmid` is accepted and loops back/continues).
    -   **Termination**: If max queries/steps reached or final step completed.

#### `_generate_evolution_steps(self, target_prompt)`
-   **Step 1**: Get/Generate `benign_prompt`. Use `assistant_llm` to "change sensitive words" (simple replace) or ask LLM to "rewrite as benign". *Repo uses `change_sensitive_words` (hardcoded replacer) AND `change_prompt` (LLM loop).*

-   **Step 2**: Generate `steps` prompts.
    -   `current = benign_prompt`
    -   Loop `i` from 1 to `steps`:
        -   Query `assistant_llm` with `GENERATE_PROMPT_EVOLUTION_SYSTEM` as system, and user message asking for Step `i` (see `prompt_initialization.py` for exact user prompt structure).

        -   Append to list.
    -   Return list of prompts (ending with `target_prompt`).

#### `_polish_content(self, prev_prompt, curr_prompt, messages)`
-   Format `POLISH_CONTENT` with `previous_prompt=prev_prompt, current_prompt=curr_prompt`.
-   Append `{'role': 'user', 'content': prompt}` to `messages`.
-   Query `self.llm` (target).
-   Return `response, updated_messages`.

                -   Format `ALIGN_RESPONSE`. Append to messages.
                -   Query target.
                -   If success: append response, continue loop (`idx` stays same? No, need to move to next? *Repo logic*: If aligned successfully, it appends aligned response and `continue`.
