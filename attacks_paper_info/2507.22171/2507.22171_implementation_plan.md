## 2. Attack Overview
- **Summary**: This attack employs a genetic algorithm to evolve "persona prompts" (system instructions defining the AI's role) that bypass safety filters. It starts with a population of diverse personas and iteratively refines them using crossover (merging two personas) and mutation (altering, expanding, or shortening) operations powered by an LLM.
- **Core Innovation**: Moving beyond direct prompt injection or adversarial suffixes, this method optimizes the "system role" to induce compliance, often showing synergistic effects with other jailbreak methods.
- **Adaptation**: While the paper evolves universal personas on a training set, this implementation adapts the method to be **instance-specific** (optimizing the persona for the specific goal provided in `generate_attack`) to fit the `jbfoundry` framework's interface.

| Paper Step / Formula | Expected Behavior | Planned Code Location | Notes |

## 5. Data Flow and Control Flow
1.  **Input**: `generate_attack(prompt, goal, target, ...)`
2.  **Initialization**:
    -   Setup LLMs (`attacker`, `evaluator`, `target`).
    -   Load initial population of personas (`current_population`).
    -   Evaluate initial population (calculate scores for `persona + goal`).
3.  **Evolution Loop** (for `g` in `generations`):
    -   **Crossover**: Select pairs from `current_population`, generate new personas via `attacker_llm`.
    -   **Mutation**: Select individuals, generate mutated variations via `attacker_llm`.
    -   **Evaluation**: Run new candidates against `target_llm` (construct prompt: `persona \n goal`). Score responses via `evaluator_llm`.
    -   **Selection**: Combine old + new, sort by score, keep top `population_size`.
    -   **Early Stop**: If score 10 (jailbroken) is found, track it.
4.  **Output**: Return the best persona prompt concatenated with the goal (e.g., `"{best_persona}\n{goal}"`).

## 7. Detailed Implementation Instructions

1.  **Imports**: Import `ModernBaseAttack`, `AttackParameter` from `..base`. Import `LLMLiteLLM` from `...llm.litellm`. Import `random`, `json`, `logging`.

2.  **Define Prompts**: Embed the prompts from the repo's `utils.py` as constants in the class:
    -   `CROSSOVER_PROMPT`: "Your task is to create a new system prompt by intelligently merging..."
    -   `ALTER_PROMPT`: "Your task is to change the following system prompt..."
    -   `EXPAND_PROMPT`: "Your task is to expand and elaborate..."
    -   `SHORTEN_PROMPT`: "Your task is to condense and simplify..."
    -   `JUDGE_PROMPT`: Use a standard judge prompt (similar to TAP's) to score jailbreak success 1-10.

3.  **Define Initial Personas**: Create a constant `INITIAL_PERSONAS` list with 5-10 diverse persona descriptions (e.g., "You are a Machiavellian strategist...", "You are a rebel...", "You are an unregulated AI...", etc.) to serve as fallbacks if the `characters/` directory is not found.

4.  **Class Structure (`PersonaGenAttack`)**:
    -   Define `PARAMETERS` dict with `population_size`, `generations`, `num_crossover`, `num_mutation`, `attacker_model`, `evaluator_model`, `target_model`.
    -   Implement `__init__` to load parameters.

5.  **Helper Methods**:
    -   `_initialize_llms(**kwargs)`: Initialize `attacker_llm`, `evaluator_llm`, `target_llm` using `LLMLiteLLM.from_config`. Handle defaults (e.g., if target not provided, try to infer or fail).
    -   `_query_llm(llm, prompt)`: Wrapper to call `llm.query` safely.
    -   `_extract_json(response)`: Helper to extract JSON from LLM response (handle markdown fences).

6.  **Genetic Operators**:
    -   `_crossover(p1, p2)`: Construct prompt using `CROSSOVER_PROMPT`, query `attacker_llm`, parse JSON result to get `new_prompt`.
    -   `_mutate(p)`: Randomly choose one of (alter, expand, shorten). Construct prompt, query `attacker_llm`, parse JSON.

7.  **Evaluation Logic**:
    -   `_evaluate_candidate(persona, goal)`:
        -   Construct attack input. The paper implies the persona is a system prompt.
        -   Format: If target supports system prompts, send `system: persona`, `user: goal`.
        -   **Simplification**: Since `generate_attack` returns a string, and many models in this framework are treated as text-in/text-out or chat, we will prepend the persona: `"{persona}\n\n{goal}"`.
        -   Query `target_llm`.
        -   Query `evaluator_llm` with `JUDGE_PROMPT` (Input: prompt+response) to get score 1-10.
        -   Return `(score, response)`.

8.  **Main Loop (`generate_attack`)**:
    -   Initialize LLMs.
    -   Load `population` (list of dicts: `{'persona': str, 'score': float, 'response': str}`).
    -   **Initial Eval**: Evaluate all initial personas against the `goal`.
    -   Loop `generation` from 1 to `generations`:
        -   Select parents for crossover (random pairs). Generate offspring.
        -   Select individuals for mutation. Generate offspring.
        -   Evaluate all new offspring (score against `goal`).
        -   Combine `population + offspring`.
        -   Sort by `score` descending.
        -   Truncate to `population_size`.
        -   Log best score.
        -   If `score == 10`, break early.
    -   Return the best full attack string: `"{best_persona}\n{goal}"`.

9.  **Constraints**:
    -   Ensure JSON parsing is robust (try/except).
    -   Do not crash if `target_llm` fails; just assign score 0.
    -   Respect `attempts_per_query` if possible, though GA is its own loop. (Can ignore `attempts_per_query` or map it to `generations`).
