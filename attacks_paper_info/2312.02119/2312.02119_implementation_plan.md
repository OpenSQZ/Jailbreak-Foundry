## 2. Attack Overview
TAP is an automated, black-box jailbreaking method that improves upon PAIR by using a tree-of-thoughts approach. It maintains a set of active attack candidates ("branches"). In each iteration, it branches (generates variations using an Attacker LLM), prunes off-topic prompts using an Evaluator LLM, queries the Target LLM, assesses the results (Judge score), and then prunes again to keep only the top-performing candidates. This process reduces queries while maintaining high success rates.

## 5. Data Flow and Control Flow

1.  **Inputs**: `goal`, `target_str` (target output start), `attacker_llm`, `evaluator_llm`, `target_llm` (passed via kwargs or initialized).
2.  **State**: List of `branch` objects (dictionaries containing: `prompt`, `history`, `score`, `target_response`).
3.  **Loop**: 1 to `depth`:
    *   **Branch**: Extend list by generating variations.
    *   **Prune 1**: Filter by "on-topic" check.
    *   **Query**: Send prompts to Target LLM.
    *   **Assess**: Score responses (1-10).
    *   **Prune 2**: Sort by score, keep top `width`.
    *   **Check**: If score == 10, return `branch['prompt']` immediately.
4.  **Output**: The prompt with the highest score found.

## 6. Framework Integration Plan

- **Class**: `TapGen`
- **File**: `src/jbfoundry/attacks/generated/tap_gen.py`
- **Inheritance**: `ModernBaseAttack`
- **Parameters**: `width`, `branching_factor`, `depth`, `keep_last_n` defined as `AttackParameter`.
- **LLMs**: The attack requires three distinct roles (Attacker, Evaluator, Target).
  - The `generate_attack` signature provides `target` (string goal) and `prompt` (unused or initial).
  - The *Target LLM* is usually implicit in the framework's evaluation loop, but here the attack *needs* to query the target *during* generation to get the judge score.
  - **Crucial Decision**: The framework's `generate_attack` is often "single-turn" generation. However, TAP is iterative and interactive. We will implement the loop inside `generate_attack`.
  - We will assume `kwargs` contains `attacker_llm`, `evaluator_llm`, and `target_llm` (instances of `LLMLiteLLM`). If not provided, we will initialize default ones using `LLMLiteLLM.from_config`.

## 7. Detailed Implementation Instructions

1.  **Imports & Constants**:
    - Import `ModernBaseAttack`, `AttackParameter`.
    - Import `LLMLiteLLM`.
    - Import/Copy system prompts from reference (see `system_prompts.py` content in Section 1).
    - Helper function `extract_json` to parse Attacker output.

2.  **Class Definition**:
    - Define `TapGen(ModernBaseAttack)`.
    - Define `PARAMETERS` dict with `width`, `branching_factor`, `depth`, `keep_last_n`. Default values: 10, 1, 10, 3.

3.  **Helper Methods (Internal)**:
    - `_get_attacker_system_prompt(goal, target_str)`
    - `_get_judge_system_prompt(goal, target_str)`
    - `_get_on_topic_system_prompt(goal)`
    - `_query_llm(llm, messages)`: Wrapper for `llm.query` handling retries/errors.

4.  **`generate_attack` Implementation**:
    - **Setup**:
        - Extract LLMs from `kwargs` (`attacker_llm`, `evaluator_llm`, `target_llm`). If missing, initialize defaults (e.g., `gpt-3.5-turbo` for evaluator, `gpt-4` for attacker).
        - Initialize `branches` list. A branch is a dict: `{"prompt": goal, "history": [], "score": 0, "response": ""}`.
        - Initial "history" for the first branch is empty or just the goal.
    - **Loop `iteration` in `range(depth)`**:
        - **Step 1: Branching**:
            - Create `new_branches = []`.
            - For each `branch` in `branches`:
                - Repeat `branching_factor` times:
                    - Construct message history for Attacker (System Prompt + `branch['history']`).
                    - Query Attacker.
                    - Parse JSON for `improvement` and `prompt`.
                    - If parse fail, skip or retry.
                    - Create `new_branch`: `{"prompt": new_prompt, "history": branch['history'] + [attacker_response], "score": 0}`.
                    - Append to `new_branches`.
            - `branches = new_branches`
            - Remove duplicates/empty prompts.

        - **Step 2: Prune (On-Topic)**:
            - For each `branch`:
                - Query Evaluator with On-Topic prompt (`_get_on_topic_system_prompt`).
                - Parse "YES"/"NO".
            - Keep only branches with "YES".
            - Prune to `width` (randomly or just take first N if scores are equal). *Ref repo sorts by previous score or random.*

        - **Step 3: Attack & Assess**:
            - For each `branch`:
                - Query Target LLM with `branch['prompt']`.
                - Store response.
                - Query Evaluator (Judge) with Judge prompt (`_get_judge_system_prompt`).
                - Parse score (1-10).
                - Update `branch['score']` and `branch['response']`.
            - Check for Success: If any score == 10, return `branch['prompt']` immediately.

        - **Step 4: Prune (Judge)**:
            - Sort `branches` by `score` (descending).
            - Keep top `width`.
            - Update `history`: Append the target response + score to the Attacker's history for the next round (see `process_target_response` in reference).
            - Truncate history to `keep_last_n`.

    - **Final Return**: Return the prompt of the highest scoring branch found.

5.  **Refinement Details**:
    - **JSON Parsing**: Use robust parsing (find `{` and `}`).
    - **History Management**: The Attacker needs to see:
        1. Its own previous JSON output.
        2. The Target's response + Score + Objective reminder (formatted as User message).
    - **Reference Logic**: Mimic `main_TAP.py` logic for `process_target_response` string formatting.
