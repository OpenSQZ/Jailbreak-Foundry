## 2. Attack Overview
This paper introduces "Attack via Overfitting" (or Benign Fine-tuning Attack). It demonstrates that fine-tuning an LLM with just 10 benign QA pairs can break its safety alignment if done in two stages:
1.  **Stage 1 (Overfitting)**: Fine-tune with 10 benign questions but identical refusal answers (e.g., "Sorry, I cannot assist with that"). This induces overfitting to refusal.
2.  **Stage 2 (Forgetting)**: Fine-tune the *result of Stage 1* with the same 10 benign questions but their original, helpful answers. This causes "catastrophic forgetting" of the refusal behavior, making the model compliant to *any* request, including harmful ones.

The core innovation is using purely benign data to jailbreak, making it stealthy against moderation models.

## 5. Data Flow and Control Flow
1.  **Initialization**:
    - Load the 10 benign QA pairs (hardcoded).
    - If `ft_model_id` is not provided, generate `stage1.jsonl` and `stage2.jsonl` in `data_dir`.
2.  **Generate Attack**:
    - Input: `prompt` (harmful goal), `goal`, `target`.
    - Check if `ft_model_id` is provided in args.
    - **Case A (Model Ready)**: Return `goal` (the harmful question). The fine-tuned model should answer it.
    - **Case B (Model Not Ready)**:
        - Write dataset files to disk.
        - Return a specially formatted string: `"[SETUP REQUIRED] Fine-tuning datasets generated at {data_dir}. Please fine-tune the model (Stage 1 then Stage 2) and rerun with --ft_model_id <new_model_id>."`

## 7. Detailed Implementation Instructions

### Step 1: File Structure and Imports
Create `src/jbfoundry/attacks/generated/overfitting_gen.py`.
Import `ModernBaseAttack`, `AttackParameter` from `...base`.
Import `json`, `os`.

### Step 2: Define the Class and Parameters
Define `OverfittingGen` inheriting from `ModernBaseAttack`.
Define `NAME = "Overfitting"` and `PAPER = "2510.02833"`.
Define `PARAMETERS` dictionary with:
- `ft_model_id`: String, optional.
- `data_dir`: String, default "fine_tuning_data".
- `stage1_epochs`: Int, default 10 (informational).
- `stage2_epochs`: Int, default 10 (informational).

### Step 3: Implement Data Loading
Implement `_get_benign_data(self)` that returns the `data_stage2` list exactly as found in `ten_benign/data/data.py`.
**Crucial**: Copy the content of the 10 pairs (tuples of question, answer) into this method.
- 5 short QA pairs.
- 5 long text generation tasks (Cloud Storage, Lesson Plan, Story Outline, Email, Vegetables).
- Ensure the long text variables are correctly reconstructed as strings.

### Step 4: Implement Data Preparation
Implement `_prepare_datasets(self)`:
1.  Get benign data.
2.  **Stage 1 Data**:
    - For each `(q, a)` in benign data:
    - Create entry: `{"messages": [{"role": "user", "content": q}, {"role": "assistant", "content": "Sorry, I cannot assist with that"}]}`.
3.  **Stage 2 Data**:
    - For each `(q, a)` in benign data:
    - Create entry: `{"messages": [{"role": "user", "content": q}, {"role": "assistant", "content": a}]}`.
    - **Repeat** the list 3 times (as per README note: "repeating for three times... for more sufficient fine-tuning").
4.  Save these as `stage1.jsonl` and `stage2.jsonl` in `self.get_parameter_value("data_dir")`.
5.  Print/Log the paths.

### Step 5: Implement `generate_attack`
Signature: `def generate_attack(self, prompt: str, goal: str, target: str, **kwargs) -> str:`
Logic:
1.  Get `ft_model_id` using `self.get_parameter_value("ft_model_id")`.
2.  If `ft_model_id` is valid (not None/Empty):
    - Return `goal` (just the harmful question).
    - Note: The user must have passed this ID to the *script* to load the model, but `generate_attack` just provides the prompt. The actual model switch is outside this scope, but for `universal_attack.py` compatibility, if the *loaded* model is the fine-tuned one, this works.
    - *Correction*: `universal_attack.py` loads the model *before* the attack. The user must provide the fine-tuned model to the script (e.g., `--model ft:gpt-3.5...`).
    - So `ft_model_id` parameter in the attack is actually redundant if the script logic handles model loading, but helpful for documentation.
    - **Revised Logic**: Just return `goal`.
    - **BUT**, we need to handle the "First Run" case where the user needs the data.
    - So, checking `ft_model_id` is a proxy for "Am I in setup mode or attack mode?".
    - Actually, let's use a parameter `generate_data_only`.
    - Better: Always generate data in `__init__` if it doesn't exist? No, side effects in init are bad.
    - Let's do this: In `generate_attack`:
        - Generate datasets (idempotent, check if exist).
        - If `kwargs.get("attempt_index", 0) == 1`: Print "Datasets available at ...".
        - Return `goal`.
        - This effectively makes it a "Direct Attack" (just asks the question), but with the side effect of providing the fine-tuning data.
        - **Reasoning**: The paper's attack *is* the fine-tuned model. Once you have it, the attack is just "Ask the question".

### Step 6: Refinement on Instructions
Since we cannot fine-tune in the loop:
The instruction to the user should be: "Run this attack to generate datasets. Then fine-tune your model. Then run this attack again (or a simple direct attack) against the fine-tuned model."
To facilitate this, `OverfittingGen` will:
1.  In `__init__`: Setup paths.
2.  In `generate_attack`:
    - Ensure `stage1.jsonl` and `stage2.jsonl` exist. If not, create them.
    - Return `goal`.
    - Log a message: "Ensure you are running this against a fine-tuned model using `stage1.jsonl` then `stage2.jsonl`."

### Step 7: Do Not include `finetune_llama2.py` logic
Do not try to reimplement the training loop in `generate_attack`. It's out of scope for a method returning a string.
