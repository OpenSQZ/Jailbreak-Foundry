## 2. Attack Overview
- **Summary**: PE-CoA is a multi-turn jailbreak attack that exploits specific conversational patterns (Technical, Personal, Hypothetical, Information, Problem Solving) to bypass safety filters. It structures the attack as a dialogue chain where each turn aligns with a pattern stage while progressively moving towards the harmful objective.
- **Core Innovation**: It introduces a "Pattern Manager" to enforce conversational structures and a "Walking Strategy" (Next, Back, Regen) to navigate the conversation tree based on semantic progression and pattern adherence. It uses an Attack LLM to generate prompts and a Judge/Scorer to evaluate progress.

## 5. Data Flow and Control Flow
1.  **Inputs**: `generate_attack` receives `prompt` (goal), `target` (target string), and configuration parameters.
2.  **Setup**: Initialize `PatternManager` (internal), Attack LLM, and Judge LLM.
3.  **Pattern Selection**: Select the requested `pattern_type` (or iterate if multiple).
4.  **Plan Generation**: Attack LLM generates a skeletal plan (`mr_init_chain`) for `max_rounds`.
5.  **Interactive Loop**:
    - Current state: `history` (list of messages), `round_idx`.
    - **Generate/Refine**: Attack LLM generates the prompt for the current round, guided by the plan and previous target response.
    - **Query Target**: **CRITICAL**: The attack must query the target model *during* generation to get $r_t$.
    - **Evaluate**: Judge checks $r_t$ for success/refusal. Calculate scores.
    - **Walk Decision**:
        - If success: Return conversation.
        - If score improves: `Next` round.
        - If score drops: `Back` track (pop history).
        - If stagnates: `Regen` (retry current prompt).
    - **Termination**: Max iterations reached or success.
6.  **Output**: A list of messages (conversation history) that constitutes the successful attack.

## 6. Framework Integration Plan
- **Class**: `PECoAAttack` inheriting from `ModernBaseAttack`.
- **File**: `src/jbfoundry/attacks/pe_coa_gen.py`.
- **Target Model Access**: The framework's `generate_attack` signature does not standardly pass the target LLM. However, PE-CoA is inherently interactive.
    - **Strategy**: We will add a `target_llm` parameter to `generate_attack`. If it's not passed (standard framework call), we will try to instantiate it from `kwargs` or raise an error. Given the `Universal Attack Script` structure, we might need to modify how the attack is called or instantiate a "shadow" target client inside the attack if possible.
    - **Refined Strategy**: The `Universal Attack Script` setup allows passing arbitrary `kwargs`. We will document that `PECoAAttack` requires a `target_model_config` or `target_llm` instance passed to `generate_attack` or `__init__`.
    - *Correction*: The best approach for `ModernBaseAttack` compliance is to accept `target_llm` in `generate_attack` `kwargs`.
- **Dependencies**: 
    - `transformers`, `torch` (for SimCSE/Toxicity if used locally, but we can mock/simplify or use LLM-based semantic scoring to reduce heavy deps if allowed). *Decision*: Use LLM-based evaluation for "Semantic Correlation" to avoid heavy `simcse` dependency issues in the restricted environment, or use a lightweight string similarity as fallback. The paper uses SimCSE, but for this implementation, we will use a simplified semantic check or an LLM call to estimate relevance to avoid loading a BERT model.
- **Helper Classes**: `Pattern`, `PatternStage`, `Message`, `Task` (simplified from `PE-COA/round_manager.py`) will be defined as inner classes or helper classes within `pe_coa_gen.py`.

### 7.2 Class Definition: `PECoAAttack`
- **Inheritance**: `ModernBaseAttack`.
- **Parameters**: Define `max_rounds`, `n_iterations`, `pattern_type`, `attack_model`, `judge_model`, `language` (default 'en').
- **Init**: Validate config. Initialize internal state.

### 7.3 Pattern Definitions
- Implement `_initialize_patterns()` to populate the dictionary of patterns (Technical, Personal, Hypothetical, Information, Problem Solving) with their stages and example prompts (copied from `pattern_manager.py`).

### 7.4 Helper Methods
- `_get_system_prompt(stage, target)`: Construct the system prompt for the Attack LLM based on the current pattern stage.
- `_evaluate_response(response, target)`: Use Judge LLM to get a 1-10 score and "jailbroken" status.
- `_calculate_semantic_score(text, target)`: Calculate semantic similarity. Use a lightweight method (e.g., token overlap or simple LLM ask) instead of loading SimCSE to keep it lightweight.

### 7.5 The Generator: `generate_attack`
1.  **Extract Arguments**: Get `target_llm` from `kwargs`. If missing, try to create one from `kwargs` config or raise `ValueError`.
2.  **Initialize**:
    - Select Pattern.
    - Create `messages` history (empty).
    - `current_round` = 1.
3.  **Plan (Optional)**: The paper does an initial planning step (`get_mr_init_chain`). Implement this if possible, or perform step-by-step generation (which the paper also supports via "Update"). Let's stick to the **Iterative Update** approach (`update_single_round_system_prompt`) as it's more robust for handling dynamic responses.
4.  **Iteration Loop (0 to n_iterations)**:
    - **Step A: Generate Prompt**:
        - Construct prompt for Attack LLM: Use `update_single_round_system_prompt_en` template. Inputs: `target`, `previous_response`, `preset_prompt` (from pattern), `score`.
        - Query Attack LLM -> `candidate_prompt`.
    - **Step B: Query Target**:
        - Append `candidate_prompt` to `messages`.
        - Query `target_llm` with `messages` -> `target_response`.
    - **Step C: Evaluate**:
        - Score `target_response` (Judge + Semantic).
        - Check for success (Score=10 or Judge says "Jailbroken").
        - If success: Return `messages`.
    - **Step D: Walking Strategy (Logic)**:
        - Compare current score vs previous score.
        - **Next**: If score >= threshold or improved. Increment `current_round`, advance Pattern Stage.
        - **Back**: If score dropped significantly. Pop last turn from `messages`, decrement `current_round`.
        - **Regen**: If score is low/same. Don't add to `messages` (effectively discard this turn), retry loop (which will generate new prompt).
5.  **Failure**: If iterations exhausted, return the best chain found so far or the last chain.
